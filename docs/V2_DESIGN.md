# 데이터 수집·저장 설계

## 무엇을 수집했는가

데이터 원천은 한국어 위키피디아이며, 수집 범위는 두 개의 분류 트리다. "분류:대한민국의 영화"와 "분류:대한민국의 배우"를 각각 하위 분류 깊이 3까지 순회해 문서 목록을 얻었고, 두 트리에서 발견된 문서는 각각 16,149건과 8,904건이었다. 배우 문서의 상당수(5,414건)는 영화 분류 트리에도 함께 속해 있어 제목 기준으로 중복을 제거했으며, 최종 수집 대상은 합집합 19,639개 문서다. 배우 트리를 별도로 포함한 이유는 서비스가 영화 자체에 대한 질문("이 영화의 내용은?")뿐 아니라 인물 중심 질문("이 배우가 출연한 영화는?")에도 답해야 하기 때문으로, 인물 정보의 원천을 영화 문서에 딸린 언급이 아니라 인물 문서 자체에서 확보하기 위함이다.

각 문서에서는 서두만이 아니라 문서 전체를 수집했다. 위키 문서의 서두는 요약 성격이라 정보 밀도는 높지만, "줄거리를 자세히 알려달라"거나 "평가는 어땠는가" 같은 질문에 답할 세부 내용은 본문 섹션에 있다. 사전 표본 조사(무작위 100편)에서 문서당 본문 길이 중앙값은 영화 약 3,800자·배우 약 2,400자, 섹션 수 중앙값은 5~6개로 확인되어, 서두 대비 수 배의 지식이 본문에 존재함을 실측으로 확인하고 전문 수집을 결정했다. 다만 내용이 사실상 없는 스텁 문서는 품질 기준(서두 길이 하한)으로 걸러냈다. 이미지는 파일 자체가 아니라 캡션 등 설명 텍스트만 취했고, 복잡한 표는 완전한 구조 해석 대신 행 단위 텍스트로 변환해 담았다.

## 어떻게 쪼개어 저장했는가 — 섹션 단위 청크

검색의 기본 단위인 청크는 문서 전체도, 문장도 아닌 "섹션"으로 잡았다. 문서 전체를 하나의 벡터로 만들면 여러 주제가 한 벡터에 눌려 담겨 어떤 세부 질문과도 뚜렷하게 맞지 않는 흐릿한 표현이 되고, 문장 단위로 자르면 맥락이 사라져 벡터가 불안정해진다. 위키 문서의 섹션(줄거리, 출연진, 평가, 수상 등)은 이미 사람이 주제 단위로 나눠 놓은 자연 경계이므로, "청크 하나 = 한 덩어리 주제"라는 원칙에 가장 부합하는 단위다.

청크 식별자는 문서 제목과 섹션명을 결합한 계층형 규칙을 썼다. 예컨대 「극한직업」 문서는 극한직업::서두, 극한직업::줄거리, 극한직업::출연진과 같은 청크들로, 「유해진」 문서는 유해진::서두, 유해진::출연 작품과 같은 청크들로 저장된다. 섹션 길이는 문서마다 제각각이지만 청크 크기를 인위적으로 통일하지는 않았다. 임베딩 모델은 입력 길이와 무관하게 고정 차원 벡터를 출력하므로 크기 차이 자체는 검색에 문제가 되지 않으며, 문제가 되는 것은 양 극단뿐이다. 그래서 지나치게 긴 섹션은 문단 경계에서 나누어 극한직업::줄거리::1처럼 하위 번호를 붙였고, 몇 줄짜리 잔섹션은 상위 섹션에 병합했으며, 하한 미달 조각은 제외했다. 즉 "동일 크기"가 아니라 "허용 대역 안의 자연스러운 크기"로 정돈한 것이다.

모든 청크에는 검색 필터링을 위한 메타데이터를 부여했다. 문서 제목, 문서 유형(영화/인물 — 위키 분류 태그로 자동 판정), 섹션명, 그리고 해당 문서에 붙은 위키 분류 목록("2019년 영화", "대한민국의 코미디 영화" 등)이 그것이다. 이 메타데이터 덕에 "영화 문서의 줄거리 섹션에서만 검색"처럼 범위를 좁힌 조회나, 임베딩 없이 분류 태그로 곧바로 후보를 모으는 조회가 가능해진다. 벡터 저장소는 ChromaDB를 사용하고 유사도는 cosine distance로 명시 설정했으며, 수집 원문 스냅샷은 별도 파일로 보존해 재현성을 확보했다 — 위키 원문은 계속 수정되므로 스냅샷이 없으면 데이터의 성립 근거가 사라지기 때문이다.

## 벡터 검색만으로 부족한 질문을 위한 정형 지식층

문서 표본 조사에서 확인된 중요한 사실 하나는, 조사한 모든 문서에 인포박스(정보 상자)가 존재한다는 점이었다. 인포박스에는 영화라면 감독·개봉일·장르·출연진이, 인물이라면 출생·데뷔 정보가 반정형으로 담겨 있다. 이를 파싱해 문서별 정형 레코드로 별도 저장했다. "개봉일이 언제인가" 같은 단일 사실 질문은 의미 검색보다 이 레코드 조회가 더 빠르고 정확하며, 답의 근거가 명시적이라는 장점이 있다.

또한 출연 관계를 양방향으로 조회할 수 있는 필모그래피 색인을 구축했다. 배우 문서의 출연 작품 섹션에서 1차로 추출하고, 여기에 모든 영화 인포박스의 출연진 필드를 역방향으로 뒤집은 결과("극한직업의 출연: 류승룡" → "류승룡의 출연작: 극한직업")를 합쳐 보완했다. 출연작 섹션이 없는 배우 문서가 일부 있지만 인포박스는 전 문서에 존재하므로, 두 원천을 합치면 출연 관계망이 상당히 완전해진다. 이런 목록형 질문은 의미가 비슷한 문서 몇 개를 찾는 문제가 아니라 관계를 빠짐없이 집계하는 문제라서, 상위 k개만 돌려주는 벡터 검색으로는 완전성을 보장할 수 없고 명시적 관계의 집계가 정공법이기 때문이다.

구체적인 형태를 예시로 보이면 다음과 같다(값은 수집 시점의 위키 원문 스냅샷을 따른다). 먼저 인포박스 정형 레코드는 문서 하나당 하나씩 만들어지는 키-값 묶음으로, 영화 문서와 인물 문서가 서로 다른 필드 구성을 갖는다.

```json
{ "title": "극한직업", "doc_type": "movie",
  "감독": "이병헌", "각본": "배세영", "개봉일": "2019-01-23",
  "장르": "코미디", "출연": ["류승룡", "이하늬", "진선규", "이동휘", "공명"],
  "관객수": "16,265,618명",
  "source_chunk": "극한직업::서두" }

{ "title": "유해진", "doc_type": "person",
  "출생": "1970-01-04", "직업": "배우", "데뷔": "1997년",
  "source_chunk": "유해진::서두" }
```

"극한직업의 감독은?"이나 "유해진은 언제 태어났나?" 같은 사실 질문은 이 레코드의 해당 필드를 읽는 것으로 끝나며, source_chunk 필드가 원본 청크를 가리키고 있어 어느 문서에 근거한 값인지 즉시 역추적된다.

다음으로 필모그래피 색인은 인물 이름을 키로 출연 관계 전체를 담는 목록이며, 각 항목에 어느 원천에서 왔는지가 함께 기록된다 — 배우 문서의 출연 작품 섹션에서 직접 추출된 항목과, 영화 인포박스의 출연진 필드를 뒤집어 보충된 항목이 한 목록에 합쳐진 모습이다.

```json
"유해진": [
  { "작품": "택시운전사", "연도": 2017, "출처": "유해진::출연 작품" },
  { "작품": "봉오동 전투", "연도": 2019, "출처": "유해진::출연 작품" },
  { "작품": "공조", "연도": 2017, "출처": "공조::인포박스(역인덱스)" }
]
```

"유해진이 출연한 영화를 알려달라"는 질문은 이 키 하나의 조회로 목록 전량이 확보되고, 각 항목의 연도·장르 같은 부가 정보는 해당 영화의 인포박스 레코드에서 이어 붙인다. 마지막으로 분류 색인은 위키 분류 태그를 키로 소속 문서들을 묶은 것이다.

```json
"분류:대한민국의 코미디 영화": ["극한직업", "럭키", "스물", "베테랑", "..."]
```

"코미디 영화 추천"처럼 장르·범주로 후보를 모아야 하는 질문에서 임베딩 검색 대신 이 색인이 후보 집합을 제공한다. 세 구조 모두 원본 문서 식별자를 물고 있다는 공통점이 있으며, 이것이 어느 계층에서 나온 답이든 출처 표기가 일관되게 이루어지는 근거다.

## 세 층은 분담이 아니라 합성이다

정리하면 데이터는 세 가지 형태로 저장된다. 의미 검색을 위한 섹션 청크, 사실 조회를 위한 인포박스 레코드, 관계·목록 조회를 위한 필모그래피·분류 색인이다. 중요한 설계 원칙은 이 세 층이 질문 유형별 전담 창구가 아니라 하나의 질문에 함께 동원되는 재료라는 점이다. 예컨대 "이 영화의 내용을 알려달라"는 질문에는 줄거리 청크가 주 재료가 되지만 인포박스의 감독·개봉일·장르가 답변의 뼈대를 보태고 출연 관계가 맥락을 더하며, "이 배우가 나온 영화 중 가장 흥행한 작품은?" 같은 질문은 세 층의 정보가 모두 있어야만 성립한다. 같은 지식을 세 가지 형태로 겹쳐 보관함으로써 어떤 질문이 와도 최소 한 층에는 걸리고, 대개는 여러 층이 겹쳐 답을 두텁게 만드는 구조다. 각 정형 레코드와 색인 항목은 원본 청크의 식별자를 함께 보관해, 어느 층에서 나온 답이든 근거 문서로 역추적할 수 있게 했다.

물론 이 구조가 모든 것을 해결하지는 않는다. 위키에 줄거리가 작성되지 않은 영화(표본 조사 기준 약 44%)의 내용은 어떤 저장 구조로도 만들어낼 수 없고, 작품에 대한 주관적 평가는 문서에 서술된 평가를 전달하는 수준까지가 한계다. 데이터가 커버하는 범위 안에서는 최대한 넓게 답하되, 경계 밖에서는 지어내지 않고 확인 불가로 응답하는 것이 시스템의 일관된 원칙이다.

## 한 번의 hop에서 검색이 이루어지는 절차

멀티홉 질문은 hop 단위로 처리되며, 각 hop은 "현재 hop의 검색 질의" 하나를 입력으로 받는다. 이 질의에 대해 세 층이 함께 조회되지만, 층마다 조회 방식은 다르다. 세 층이 같은 방식으로 top-k를 수행하는 것이 아니라, 층의 성질에 맞는 방식이 각각 적용된다.

1층인 섹션 청크에만 유사도 기반 top-k 검색이 적용된다. 현재 hop 질의를 임베딩해 cosine 거리가 가까운 상위 k개의 청크를 가져오는 방식으로, "내용", "평가"처럼 표현이 다양해 정확한 키를 특정할 수 없는 의미 질문을 담당한다. k 값은 하이퍼파라미터로 관리한다.

2층(인포박스 레코드)과 3층(필모그래피·분류 색인)은 top-k가 아니라 키 조회 방식이다. 현재 hop 질의에서 인식된 개체명 — 영화 제목이나 인물 이름 — 을 키로 삼아, 인포박스는 해당 문서의 정형 레코드를, 필모그래피 색인은 해당 개체에 연결된 관계 목록 전체를 꺼낸다. 이 층들에 "상위 k개"라는 개념이 없는 이유는 조회의 성질이 다르기 때문이다. 레코드는 있거나 없거나 둘 중 하나이고, 출연작 목록은 일부만 가져오면 오히려 틀린 답(불완전한 목록)이 되므로 전량을 가져와야 한다. 유사도 검색은 "비슷한 것 몇 개"를 찾는 도구이고, 키 조회는 "정확히 그것"과 "빠짐없는 전부"를 꺼내는 도구라는 역할 구분이다.

세 층의 조회 결과는 하나의 검색 결과 묶음으로 병합된다. 같은 문서에서 나온 항목은 출처를 유지한 채 정리하고, 개체명이 인식되지 않았거나 해당 레코드가 없어 비어 있는 층은 자연스럽게 빠진다. 검색 품질의 문지기 판정(1등 결과조차 질의와 동떨어져 있으면 즉시 실패 처리하는 안전판)은 유사도 점수가 존재하는 1층을 기준으로 한다. 다만 1층이 거리 기준에 미달하더라도 2·3층에서 키 조회가 적중한 경우에는 hop 자체를 실패 처리하지 않는다 — 정확 일치라는 더 강한 근거가 이미 확보되어 있기 때문이다. 이때 미달 판정을 받은 1층의 청크들은 병합 묶음에서 제외하고 2·3층의 결과만 재료로 사용한다. 즉 문지기의 "차단하지 않음"은 hop의 진행 여부에 대한 것이지, 품질 미달로 판명된 유사도 검색 결과까지 재료로 쓴다는 뜻이 아니다.

이렇게 병합된 묶음이 해당 hop의 판정과 추출의 재료가 된다. 판정 단계는 이 묶음만으로 현재 hop 질의에 답할 수 있는지를 평가해 다음 단계(다음 hop으로 전환, 질의 재작성, 답변 생성)를 결정하고, 다음 hop으로 넘어가는 연쇄형 질문에서는 이 묶음에서 연결 개체(예: 영화 문서에서 찾은 감독의 이름)를 추출해 다음 hop의 검색 질의를 완성한다. 이때 인포박스 레코드에 해당 필드가 있으면 문장에서 추출하는 것보다 우선 사용한다 — 정형 필드가 자유 서술문보다 오류 여지가 작기 때문이다. 최종 답변 생성 단계 역시 마지막 hop의 묶음(연쇄형이 아닌 비교형 질문에서는 앞선 hop의 묶음까지 재조회해 포함)을 근거로 삼으며, 각 항목이 어느 층의 어느 문서에서 왔는지가 보존되어 있어 답변의 출처 표기가 층과 무관하게 일관되게 이루어진다.
